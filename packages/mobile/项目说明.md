# 移动端（`packages/mobile`）项目说明

## 定位与功能

基于 **Expo / React Native** 的「时间管家」配套客户端，与桌面端共用同一套服务端与 `@time-manger/shared` 同步协议。当前能力概览：

| 模块 | 说明 |
|------|------|
| **登录 / 登出** | 邮箱密码登录，Access Token + Refresh Token；设备维度带 `X-Device-Id`。 |
| **今日统计** | 首页展示当日时间记录汇总（应用名、时长），数据经同步写入本地缓存后读取。 |
| **日记** | 列表、编辑、删除；本地修改打脏标后 push，pull 合并服务端版本。 |
| **工作清单** | 同上：增删改、完成态；与桌面端同一 `worklist-items` 资源。 |
| **同步** | `SyncProvider` 内建 `SyncEngine`：登录后自动同步、每 5 分钟定时同步、下拉刷新触发；资源顺序为 `time-records` → `diaries` → `worklist-items`。 |

## 数据与存储（重要）

- **业务数据的权威来源是服务端**（PostgreSQL 等），移动端**不向用户承诺「仅存在手机上的第二套业务库」**；展示与编辑都应对齐账号在云端的数据。
- **实现上**：为支持增量同步与离线合并，端上使用 **SQLite（`timemanger_sync.db`）** 存放「同步缓存表」（如 `time_record_cache`、`diary_cache`、`worklist_item_cache`）以及 `sync_meta`（各资源上次同步水位）。这是**同步机制所需的缓存层**，不是另一套与接口无关的本地业务源；**登出会清空**（见 `clearSyncData`）。
- **鉴权**：Token 等放在 `expo-secure-store`（`authStore`），与业务缓存区分对待。

若产品要求「界面层只认接口、完全不落盘」，需要在架构上替换 `LocalStore` + SQLite，改为纯内存或每次进入页面直连 REST；当前仓库实现为 **SyncEngine + 本地缓存**，与桌面端一致。

## 同步水位（与「桌面新增、手机看不到」相关）

服务端 pull 使用 **`updatedAt > since`**。历史上若用响应里的**墙钟 `serverTime`** 作为 `since`，可能在同一时刻附近插入的行 **`updatedAt` 略早于该墙钟**，导致手机端**永远拉不到**该行。  
已在 **`@time-manger/shared` 的 `SyncEngine`** 中改为：仅在本次 pull **实际拉到记录**时，用记录上的 **`updatedAt` 最大值**推进 `sync_meta`；**空 pull 不推进水位**；push 成功后用 **accepted 的 `updatedAt`** 与当前水位取较大值更新。

**仍漏数据时的自查**：同一账号、手机已登录并下拉同步；服务端该行的 `userId` 与当前用户一致；桌面端是否已把该行同步到服务端（未同步则手机也看不到）。

## 开发与运行

在 monorepo 根目录（需已安装依赖）：

- `pnpm mobile:start` — 启动 Expo 开发服务（见根目录 `package.json` 脚本）。

API 基地址在 `useAuth` / 环境配置中设置（与桌面端同一后端）。
