# 桌面端数据同步链路说明

本文面向维护者，说明「桌面端如何把本地数据同步到服务端」，并标注关键文件与函数。

## 1. 一句话总览

桌面端同步采用 **事件驱动 + 离线队列**：

1. 本地数据变更先写入 `sync-state.json` 的 `dirty` 队列；
2. 主进程广播 `sync:request`；
3. 渲染进程 `SyncProvider` 收到事件后触发 `SyncEngine.syncAll()`；
4. 每个资源按「先 pull，再 push」执行；
5. 服务端在 `/api/v1/sync/:resource` 做鉴权、校验、LWW 冲突处理并写库；
6. 客户端根据 `accepted/rejected` 清理或保留本地脏记录。

## 3. 链路分层与函数定位

## A. 入口层（渲染进程）

- `src/main.jsx`
  - 根组件包裹 `<SyncProvider>`，使所有窗口页面可共享同步上下文。
- `src/sync/SyncProvider.jsx`
  - `triggerSync()`：同步触发总入口，内部调用 `engineRef.current.syncAll(RESOURCES)`。
  - `useEffect(authState...)`：当登录态可用时创建 `SyncEngine` 实例并立即首次同步。
  - `window.timeManagerAPI.sync.onRequest(...)`：监听主进程广播，触发防抖同步。
- `src/LoginWindowApp.jsx`
  - 登录成功后 `setAuthState(...)`，间接触发 `SyncProvider` 重建引擎并同步。
  - 点击「立即同步」会直接调用 `triggerSync()`。

## B. IPC 桥接层

- `preload.cjs`
  - 暴露 `window.timeManagerAPI.sync.*`：
    - `getState/setState/markClean/applyRemoteRecords`
    - `onRequest(callback)`（监听 `sync:request`）

## C. 主进程本地队列层（离线优先）

- `electron-main.js`
  - `readSyncState()/writeSyncState()`：维护 `userData/sync-state.json`。
  - `updateDirtyTimeRecords(snapshot)`：把桌面统计快照写入 `dirty['time-records']`。
  - `markDirtyDiary()` / `markDirtyWorklistItem()`：本地修改后入脏队列并广播同步请求。
  - `broadcastSyncRequest(reason)`：向各窗口发送 `sync:request`。
  - `queueLegacyDesktopContentForSync()`：兼容老数据，补齐缺失 id/updatedAt 并入队。
  - `ipcMain.handle('sync:getState'|'sync:setState'|'sync:markClean'|'sync:applyRemoteRecords')`
    - 渲染进程同步读写都通过这些 handler。

## D. 通用同步引擎层（shared 包）

- `packages/shared/src/sync/engine.ts`
  - `syncAll(resources)`：顺序同步多个资源。
  - `syncResource(resource)`：核心算法为「先 pull 全分页，再 push 本地 dirty」。
  - push 后依据服务端 `accepted/rejected` 调 `store.markClean(...)`。
- `packages/shared/src/sync/lww.ts`
  - `lwwServerPush()`：服务端冲突策略（Last-Write-Wins）。

## E. 桌面端适配实现层

- `src/sync/LocalStore.desktop.js`
  - 对接主进程 `sync-state`：
    - `getLastSyncAt/setLastSyncAt/getDirtyRecords/markClean`
    - `upsertRemote` 会把 `diaries/worklist-items` 回写到本地状态。
- `src/sync/ApiClient.js`
  - `pull()` -> `GET /api/v1/sync/:resource`
  - `push()` -> `POST /api/v1/sync/:resource`
  - `_request()` 内置 401 自动刷新 token 的重试逻辑。

## F. 服务端入库层（同步终点）

- `packages/server/src/routes/sync.ts`
  - `mountSyncRoutes(...)`
    - `GET /api/v1/sync/:resource`：按 `since + cursor` 拉增量分页。
    - `POST /api/v1/sync/:resource`：校验 `deviceId`、LWW 判定、Prisma upsert、返回 `accepted/rejected`。

---

## 4. 同步触发场景清单

- 登录成功后首次同步（`SyncProvider` 初始化引擎时触发）。
- 主进程检测到本地业务数据变化后广播 `sync:request`（日记、工作清单、时间记录）。
- 用户在登录窗口手动点击「立即同步」。

---

## 5. 数据一致性策略（当前实现）

- 资源级串行同步：`time-records -> diaries -> worklist-items`。
- 记录级冲突策略：LWW（`updatedAt` 新者胜）。
- push 仅清理服务端明确 `accepted` 的记录；`rejected(stale)` 保留在本地等待后续处理。
- `sync-state.json` 持久化使应用重启后仍可继续增量同步。
